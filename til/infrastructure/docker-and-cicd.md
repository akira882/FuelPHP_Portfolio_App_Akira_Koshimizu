# 概要
会社の技術スタック（AWS ECS, Aurora, Jenkins）を踏まえ、なぜレンタルサーバーではなくコンテナ技術(Docker)とCI/CDが必要なのかを、技術的メリットの観点から考察した。

# 1. Docker: 「環境差異」を撲滅する技術
レンタルサーバーは「完成品」の環境だが、大規模開発では以下の理由からDocker（コンテナ技術）が不可欠である。

## 導入のメリット (Why Docker?)
1.  **Environment Parity (環境の一致)**
    *   開発者のローカル環境（Mac/Windows）と、本番環境（AWS ECS）を「全く同じ構成」にできる。これにより「ローカルでは動いたのに本番で動かない」という致命的なバグを防ぐ。
2.  **Infrastructure as Code (IaC)**
    *   `Dockerfile` というコードで環境を管理できるため、インフラの設定ミスをバージョン管理(Git)できる。
3.  **Microservices Ready**
    *   `docker-compose` を用いることで、Webサーバー、DB、PHPなどを独立したコンテナとして連携させ、本番環境に近い構成をローカルで再現できる。

## ネットワークの基礎知識
*   **ポート番号**: コンテナ外部からのアクセスを適切なプロセス（Webサーバー等）に振り分けるための識別子。
    *   *例*: `localhost:8080` (ホスト) → `container:80` (Webサーバー)

# 2. Jenkins: 品質の自動化 (CI/CD)
手動でのテストやデプロイは人的ミスの温床となる。Jenkinsによる自動化は、開発サイクルを加速させるために必須である。

## 自動化フローの理解
1.  **CI (継続的インテグレーション)**
    *   GitHubへのPushを検知 → 自動ビルド → 自動テスト実行。
    *   *効果*: バグの早期発見。
2.  **CD (継続的デリバリー/デプロイ)**
    *   テスト通過後、自動的にDockerイメージを作成し、AWS環境へ反映。
    *   *効果*: リリース作業の属人化排除と、高速な機能提供。
